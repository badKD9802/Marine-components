KAMCO 그룹웨어 재구축 사업 - AI 도입
------------------------------------------------------------

기간: 2024.10 ~ 현재
소속: 애자일소다 선임연구원
역할: AI Agent 전담, RAG 참여, 문서요약 단독, LLM Fine-Tuning 전담
기술: Python, LangGraph, OpenAI Function Calling, vLLM, Milvus DB, Oracle/MariaDB
인프라: 개발 H100 8장, 운영 H100 8장 + H200 8장


(1) ReAct Agent 설계·구현 [전담]

[배경]
- 기존 구조는 LangGraph로 40개 이상 노드를 연결한 분기형 파이프라인
- 기능 추가 시 라우팅 로직이 복잡해져 유지보수 어려움
- "A 직원을 찾아서 해당 직원에게 메일 초안을 작성해줘" 같은 다중 요청 처리 불가
- 이 한계를 분석한 뒤 ReAct 패턴을 연구하여, Function Calling 모델 전환과 ReAct Agent 재설계를 직접 제안

[설계: ReAct 아키텍처]
- 실행 흐름: 사용자 질문 → LLM이 도구 선택(Reasoning) → 도구 실행(Acting) → 결과 분석 → 추가 도구 필요 시 반복 → 최종 답변 생성
- 기존 고정 라우팅(Intent 분류 → 파라미터 추출 → 검증 → API 호출)을 제거하고, LLM이 자율적으로 도구를 선택·조합하는 구조로 전환
- 최대 10회 반복 제한으로 무한 루프 방지
- Router, Parser 노드 제거하여 확장 가능한 구조 확보

[구현: 핵심 컴포넌트]
- ReAct 엔진 (react_agent.py, 695줄): LLM 호출 → 도구 실행 → 결과 분석 반복 루프. SSE 스트리밍으로 실시간 진행 상황 전송
- 도구 레지스트리 (tool_registry.py): 함수명 → 구현체 자동 매핑. 인증 컨텍스트 자동 주입. Sync/Async 자동 감지
- 도구 정의 (tool_definitions.py): OpenAI Function Calling JSON Schema로 18개 도구 파라미터 정의
- 중복 호출 방지: 해시 기반 도구 호출 중복 감지 및 차단
- HTML 자동 감지: 도구 반환값에 html_content 포함 시 LLM 거치지 않고 직접 스트리밍

[구현: 18개 업무 도구]
- 일정 관리 (4개): 조회, 등록, 수정, 삭제. 충돌 감지 및 강제 등록 플래그 포함
- 회의실 관리 (5개): 예약, 조회, 수정, 취소, 빈 회의실 검색
- 직원·사용자 (3개): 직원 검색(이름/부서/직급), 내 정보 조회, 팀원 목록
- 전자결재 (2개): 결재 양식 조회, 내 결재 목록
- 문서·메일 (2개): 메일 초안 생성, 문서 초안 작성
- 검색·번역 (2개): 사내 지식베이스 RAG 검색, 텍스트 번역

[구현: UX]
- SSE 기반 실시간 진행 UI (도구 실행 상태 전송, 미리보기 데이터 포함)
- 5건 이상 결과 → 캘린더/테이블 HTML 자동 생성, 4건 이하 → 자연어 나열
- 필수 파라미터 누락 시 역질문(Slot Filling)
- 일정, 회의실, 검색 결과 엑셀 내보내기

[구현: 그룹웨어 연동]
- Oracle DB 직접 조회: 직원 검색, 회의실 현황, 임원 일정
- GW XML API 연동: 일정 CRUD, 회의실 예약
- SLO 인증: 1회 인증 후 캐싱하여 전체 세션 재사용

[성과]
- 363개 질문(19개 카테고리, 오타·구어체·애매한 표현 포함) 자동화 테스트
- 평균 응답시간 3.9초
- 도구 추가 시 스키마 정의 + 구현체 추가만으로 확장 가능 (기존 코드 수정 불필요)


(2) 대규모 문서 기반 RAG 시스템 구축 [참여]

[설계: Context-Aware RAG 파이프라인]
- 메타데이터 활용: 제목, 등록 날짜, 결재 날짜 등을 Chunk에 추가하여 검색 성능 향상
- OCR 및 구조적 문서 처리: hwpx, PDF, 스캔 문서를 OCR로 텍스트 추출, 문서 구조(문단, 페이지)를 고려한 Chunking 전략 수립
- Chunk 컨텍스트 보강: 각 Chunk마다 LLM으로 요약(Abstract)과 예상 질문(Generated Q&A) 생성하여 벡터화. 원문과 다른 표현으로도 검색 가능하도록 설계
- 하이브리드 검색 + Reranking: 키워드 검색(Sparse)과 의미 기반 검색(Dense) 결합, Reranking으로 상위 문서 품질 보장
- 멀티 쿼리 + 청크 검증: 유사 질의 생성으로 검색 범위 확장, 답변 가능 여부 검증 후 기준 미충족 시 재검색

[담당 업무]
- 1,000만 건 문서의 Milvus 벡터 DB 적재 파이프라인 구현
- Chunk 전처리 및 메타데이터 생성 로직 개발

[성과]
- 답변 정확도 92%, 할루시네이션 통과율 91%
- Hitrate@5 91%


(3) 문서 요약·번역·회신문 초안 생성 [단독 개발]

[구현]
- 문서 유형(보고서, 이메일, 규정)별 전용 프롬프트 템플릿 개발
- 비동기 병렬 처리 적용하여 요약 속도 약 2배 개선
  - 개선 전: 5페이지 요약 17.5초
  - 개선 후: 9페이지 요약 12.3초

[성과]
- G-EVAL 품질 평가 4.3/5.0 달성


(4) 도메인 특화 LLM Fine-Tuning [전담]

[목적]
- RAG만으로는 금융·공공 도메인의 전문 용어와 업무 맥락을 충분히 반영하지 못하는 한계를 보완

[구현]
- 대상 모델: Gemma 27B
- 학습 데이터: 20만 건 KMS 청크를 Contextual Retrieval 방식으로 전처리. 각 청크에 문서 전체 맥락 요약을 추가하여 단독으로도 의미를 갖도록 보강
- 학습 형식: 질문 + 검색 청크 + 답변. <reasoning> 태그를 포함시켜 모델이 사고 과정을 단계별로 출력하도록 유도
- 평가: G-EVAL 기반 자동 평가 (일관성, 관련성, 유창성, 정확성 4개 축)

[성과]
- Fine-Tuning 전 대비 도메인 질의 답변 품질 개선
- <reasoning> 태그를 통해 답변 근거 투명성 확보

기술: Python, FastAPI, PostgreSQL, pgvector, OpenAI Embedding, Gemini 2.5 Flash, HTML/CSS/JS
인프라: Railway CI/CD 자동 배포

[배경]

[아키텍처]
- FastAPI 백엔드 5개 모듈:
  - 상품 관리: 등록/수정/삭제, 카테고리 관리, 이미지 업로드
  - AI 메일: Gemini 기반 수신 메일 분석, 회신문 초안 생성, 다국어 번역
  - RAG 채팅: pgvector 벡터 검색 + Gemini 2.5 Flash 기반 다국어(한/영/중) AI 상담
  - 고객 문의: 문의 접수, 상태 관리, 이메일 알림
  - 공개 API: 고객용 상품 조회, 검색, 상세 페이지
- 관리자 대시보드: 프레임워크 없이 HTML/CSS/JS로 직접 구현
- 고객용 홈페이지: 상품 카탈로그, 상세 페이지, AI 채팅 위젯

[RAG 파이프라인]
- 상품 등록 시 상품명, 규격, 엔진 호환 정보, 재질 등 메타데이터를 자동으로 OpenAI Embedding → pgvector에 적재
- 고객 질문에 벡터 검색으로 관련 상품을 찾고 Gemini 2.5 Flash가 다국어로 응답 생성
